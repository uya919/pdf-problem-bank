# 대규모 문서 관리 UX 리디자인 연구

> **작성일**: 2025-12-03
> **목적**: 100개 이상 문서 환경에서의 최적 UX 설계
> **가정**: 학년별 10개+ 시리즈, 시리즈당 문제/해설 쌍 = 최소 100-200개 파일

---

## 1. 규모 분석

### 1.1 예상 파일 구조

```
학원 문서 예상 규모:

┌─────────────────────────────────────────────────────────┐
│ 고등학교 1학년 (10+ 시리즈)                               │
│   ├─ 수학의바이블_문제.pdf / 수학의바이블_해설.pdf        │
│   ├─ 쎈_문제.pdf / 쎈_해설.pdf                          │
│   ├─ 개념원리_문제.pdf / 개념원리_해설.pdf               │
│   ├─ 기출문제집_문제.pdf / 기출문제집_해설.pdf           │
│   ├─ 블랙라벨_문제.pdf / 블랙라벨_해설.pdf               │
│   └─ ... (10개 이상)                                    │
├─────────────────────────────────────────────────────────┤
│ 고등학교 2학년 (10+ 시리즈)                               │
│   └─ ... (동일 패턴)                                    │
├─────────────────────────────────────────────────────────┤
│ 고등학교 3학년 (10+ 시리즈)                               │
│   └─ ... (동일 패턴)                                    │
├─────────────────────────────────────────────────────────┤
│ 중학교 1-3학년                                           │
│   └─ ... (각 학년 10+ 시리즈)                            │
└─────────────────────────────────────────────────────────┘

총 예상: 6학년 × 10시리즈 × 2(문제/해설) = 120개+ 파일
```

### 1.2 사용자 시나리오

| 시나리오 | 빈도 | 현재 고통점 |
|----------|------|-------------|
| "고1 수학의바이블 문제 선택" | 매우 높음 | 120개 중 1개 찾기 |
| "지난번 작업하던 파일 재선택" | 높음 | 기억에 의존 |
| "새로 업로드한 파일 선택" | 중간 | 목록 끝에 있음 |
| "비슷한 이름의 파일 구분" | 중간 | 고1쎈 vs 고2쎈 |

### 1.3 핵심 설계 원칙

```
┌───────────────────────────────────────────────────────────┐
│  원칙 1: 3초 규칙                                          │
│  → 어떤 문서든 3초 내에 찾을 수 있어야 함                   │
├───────────────────────────────────────────────────────────┤
│  원칙 2: 제로 러닝 커브                                    │
│  → 처음 사용자도 즉시 이해 가능                            │
├───────────────────────────────────────────────────────────┤
│  원칙 3: 점진적 복잡성                                     │
│  → 단순 작업은 단순하게, 복잡한 작업만 복잡하게            │
├───────────────────────────────────────────────────────────┤
│  원칙 4: 컨텍스트 인식                                     │
│  → 사용 패턴을 학습하여 예측                               │
└───────────────────────────────────────────────────────────┘
```

## 2. 근본적 재설계: 2단계 선택 패러다임

### 2.1 핵심 인사이트

> "100개의 파일에서 1개를 선택하는 것"보다
> "6개 학년에서 1개 선택 → 10개 시리즈에서 1개 선택"이 더 쉽다

**인지 부하 비교**:
- 단일 선택: 100개 스캔 → 높은 부하
- 2단계 선택: 6개 + 10개 스캔 → 낮은 부하

### 2.2 새로운 선택 플로우

## 4. 파일 네이밍 컨벤션 및 자동 파싱

### 4.1 권장 네이밍 규칙

```
{학년}_{시리즈명}_{타입}.pdf

예시:
- 고1_수학의바이블_문제.pdf
- 고1_수학의바이블_해설.pdf
- 중3_쎈_문제.pdf
- 중3_쎈_해설.pdf
```

### 4.2 자동 파싱 로직

```typescript
interface ParsedDocument {
  grade: string;      // "고1", "중3", etc.
  series: string;     // "수학의바이블", "쎈", etc.
  type: 'problem' | 'solution';
  original: string;   // 원본 파일명
}

function parseDocumentName(filename: string): ParsedDocument | null {
  // 패턴 1: {학년}_{시리즈}_{타입}.pdf
  const pattern1 = /^(중[1-3]|고[1-3])_(.+)_(문제|해설)\.pdf$/;

  // 패턴 2: {학년}{시리즈}{타입}.pdf (언더스코어 없음)
  const pattern2 = /^(중[1-3]|고[1-3])(.+)(문제|해설)\.pdf$/;

  // 패턴 3: 자유 형식 (키워드 기반)
  // "고등학교1학년" → "고1"
  // "문제집", "problem" → "문제"
  // "해설집", "solution", "답" → "해설"

  let match = filename.match(pattern1);
  if (match) {
    return {
      grade: match[1],
      series: match[2],
      type: match[3] === '문제' ? 'problem' : 'solution',
      original: filename
    };
  }

  // 추가 패턴 매칭...
  return null;
}
```

## 5. 데이터 구조 설계

### 5.1 문서 메타데이터

```typescript
interface DocumentMetadata {
  id: string;                    // 고유 ID
  filename: string;              // 원본 파일명

  // 파싱된 정보
  grade: string | null;          // "고1", "중3", etc.
  series: string | null;         // "수학의바이블", etc.
  type: 'problem' | 'solution' | null;

  // 페어링 정보
  pairedDocumentId: string | null;  // 연결된 문제/해설 문서

  // 사용 통계
  lastUsedAt: Date | null;
  useCount: number;

  // 시스템 정보
  uploadedAt: Date;
  pageCount: number;
}
```

## 11. 결론

### 핵심 변경사항

| 항목 | Before (드롭다운) | After (2단계 선택) |
|------|-------------------|-------------------|
| 선택 방식 | 100개 중 1개 선택 | 6개 → 10개 선택 |
| 인지 부하 | 높음 | 낮음 |
| 탐색 시간 | 15초+ | 3초 |
| 오류 가능성 | 높음 | 낮음 |
| 확장성 | 낮음 | 높음 |

### 핵심 인사이트

> **"드롭다운을 개선하는 것"이 아니라**
> **"문제의 구조를 재정의하는 것"이 해답**

100개의 문서는 "100개의 선택지"가 아니라
"6개 학년 × 10개 시리즈"의 구조화된 데이터

### 즉시 실행 항목

1. **파일 네이밍 가이드 제공**
   - `{학년}_{시리즈}_{타입}.pdf` 형식 권장

2. **2단계 선택 UI 구현**
   - 학년 → 시리즈 선택 플로우

3. **자동 페어링 구현**
   - 문제/해설 자동 매칭
